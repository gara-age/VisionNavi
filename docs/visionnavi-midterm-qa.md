# VisionNavi 중간발표 예상 질문 모범답변

이 문서는 VisionNavi 중간발표에서 나올 가능성이 높은 질문에 대해, 발표자가 짧고 명확하게 답할 수 있도록 정리한 Q&A 문서입니다.  
답변은 현재 프로젝트의 실제 구현 상태를 기준으로 작성했으며, 과장보다는 “지금 되는 것 / 아직 개선 중인 것”을 분명히 구분하는 방향으로 정리했습니다.

---

## 1. 기획능력 관련 질문

### 1. 이 프로젝트의 핵심 문제 정의는 무엇인가요?
고령자가 웹과 PC에서 하고 싶은 일을 알고 있어도, 실제로 그 일을 끝까지 수행하는 과정이 너무 복잡하다는 점입니다. VisionNavi는 “질문에 답하는 AI”가 아니라 “목표를 실제 화면에서 수행해 주는 AI”를 목표로 합니다.

### 2. 단순히 “고령자가 디지털 기기를 어려워한다”가 아니라, 어떤 구체적인 상황을 해결하려는 건가요?
예를 들어 유튜브에서 원하는 영상을 찾거나, 복지 정보를 검색하거나, 메모장에 내용을 적는 일처럼 여러 단계를 거쳐야 하는 작업을 쉽게 만들고자 합니다. 즉, 검색어 입력, 사이트 이동, 버튼 선택 같은 중간 절차를 줄여 주는 것이 핵심입니다.

### 3. 왜 타겟을 일반 사용자가 아니라 한국·일본 고령자로 설정했나요?
이 프로젝트의 사회적 가치가 가장 크게 드러나는 대상이 디지털 취약 고령자라고 판단했기 때문입니다. 또 한국과 일본은 고령화 속도가 빠르고, 고령자 디지털 접근성 문제를 공통으로 안고 있어 확장성이 있다고 봤습니다.

### 4. 고령자 중에서도 어떤 유형의 사용자를 우선 대상으로 삼고 있나요? 독거노인, 치매 초기 고령자, 디지털 취약 고령자 중 어디에 더 가깝나요?
현재는 치매 환자보다는 디지털 취약 고령자에 더 가깝습니다. 혼자 생활하시거나, 스마트폰과 PC 사용은 가능하지만 검색과 설정 같은 복잡한 작업을 어려워하는 분들을 우선 사용자로 보고 있습니다.

### 5. 초기 계획서에서는 취미·체험형 솔루션이 강조되었는데, 현재 구현 방향도 그 목적과 일치하나요?
네, 다만 지금은 기반 기능을 먼저 만들고 있습니다. 트로트 영상 검색, 길찾기, 복지 정보 탐색처럼 실제 생활과 여가 활동을 연결하는 기능이 그 방향과 이어집니다.

### 6. 프로젝트가 “정서적 안정”과 “사회적 고립 완화”에 실제로 어떻게 기여할 수 있나요?
혼자서 해결하기 어려운 디지털 작업을 스스로 해낼 수 있게 되면 무력감이 줄어듭니다. 또 영상 탐색, 복지 프로그램 탐색, 정보 접근이 쉬워지면 일상 활동 참여가 늘어나 사회적 고립 완화에도 기여할 수 있습니다.

### 7. 고령자의 디지털 문제를 해결하는 기존 서비스와 Navi의 가장 큰 차별점은 무엇인가요?
기존 서비스가 “답변 제공”에 머무는 경우가 많다면, VisionNavi는 실제 브라우저와 데스크톱 앱을 조작해 작업 수행까지 연결하려는 점이 다릅니다. 즉, 정보 안내를 넘어 실행까지 하는 구조입니다.

### 8. 단순 음성비서, 챗봇, 화면 낭독기, 원격지원 서비스와 비교했을 때 Navi가 더 나은 점은 무엇인가요?
음성비서와 챗봇은 답변은 잘하지만 실제 웹페이지를 대신 조작하지는 않습니다. 원격지원은 사람의 도움이 필요하지만, VisionNavi는 사용자가 직접 말하거나 입력하면 반복적인 디지털 작업을 스스로 진행할 수 있게 돕습니다.

### 9. 왜 스마트폰 앱이 아니라 Windows 데스크톱 앱 형태를 선택했나요?
현재 프로젝트는 웹 탐색과 Windows 앱 자동화를 함께 다루고 있기 때문에 데스크톱이 기술적으로 더 적합했습니다. 특히 브라우저 자동화와 메모장 같은 데스크톱 작업을 한 플랫폼에서 검증하기 좋았습니다.

### 10. 고령자에게 PC 기반 서비스가 스마트폰보다 더 적합하다고 볼 근거가 있나요?
모든 고령자에게 PC가 더 적합하다고 보지는 않습니다. 다만 큰 화면, 명확한 버튼, 브라우저와 데스크톱 작업을 직접 연결하는 데서는 PC가 실험과 초기 서비스 검증에 유리하다고 판단했습니다.

### 11. 발표자료에서는 웹 자동화와 Windows 작업 자동화를 모두 제시했는데, 중간발표 기준으로 가장 핵심 기능은 무엇인가요?
중간발표 기준 핵심은 “자연어 명령을 받아 실제 작업으로 이어지는 흐름” 자체입니다. 그 대표 시나리오가 검색·읽기와 메모장 자동화입니다.

### 12. 프로젝트의 범위가 너무 넓어질 위험은 없나요?
그 위험은 분명히 있습니다. 그래서 현재는 대표 시나리오를 검색/읽기, 메모장 작성, 일부 길찾기로 제한하고, 나머지는 장기 목표로 분리해 관리하고 있습니다.

### 13. 현재 구현 범위와 최종 목표 범위는 어떻게 구분하고 있나요?
현재 구현 범위는 검색/읽기, 메모장 입력, STT/TTS, wakeword, 고령자 친화 UI입니다. 최종 목표는 BrowserUse/ComputerUse류 외부 agent와 결합한 범용 작업 지원까지 확장하는 것입니다.

### 14. “고령자 친화 UI”를 단순히 글씨를 크게 하는 것 이상으로 어떻게 정의하고 있나요?
행동 우선순위를 명확히 보여주고, 버튼 수를 줄이고, 생활형 문구를 사용하며, 진행 상태를 쉽게 이해할 수 있게 만드는 것입니다. 즉 “보기 쉬움”뿐 아니라 “무엇을 해야 할지 바로 아는 UI”를 지향합니다.

### 15. 한국 고령자와 일본 고령자의 사용 습관 차이는 어떻게 반영할 계획인가요?
언어만 번역하는 수준이 아니라, 검색 서비스 선호, 표현 방식, 존댓말 톤, 안내 문구까지 분리할 계획입니다. 현재도 한국어/일본어 UI, STT, wakeword, 모델 라우팅을 나누는 방향으로 가고 있습니다.

## 2. 사용자 시나리오 관련 질문

### 16. 대표 사용자 시나리오를 하나만 든다면 무엇인가요?
“유튜브에서 트로트 영상 찾아줘” 같은 명령입니다. 사용자는 말만 하고, 시스템은 브라우저를 열고 검색하고 결과를 보여주는 흐름을 수행합니다.

### 17. “트로트 영상 틀어줘”라는 명령이 들어왔을 때 시스템은 어떤 순서로 동작하나요?
먼저 명령을 이해해 검색 intent로 분류하고, 유튜브가 목적지인지 확인합니다. 그 뒤 브라우저 실행, 검색어 입력, 결과 확인, 요약 또는 재생 후보 안내 순으로 진행합니다.

### 18. “근처 복지관 프로그램 검색해줘” 같은 명령은 현재 구현 가능한가요, 아니면 향후 목표인가요?
현재는 일반 웹 검색 흐름으로 부분적으로는 가능하지만, 복지관 데이터에 특화된 정확한 서비스는 아직 향후 목표입니다. 앞으로는 검색/API/RAG 기반의 검증된 정보 흐름으로 분리하려고 합니다.

### 19. 사용자가 명령을 애매하게 말하면 어떻게 처리하나요?
현재도 일부는 되묻기나 보정이 가능하지만, 아직 완전하지는 않습니다. 앞으로는 ambiguity가 높을 때 바로 실행하지 않고 “혹시 이런 뜻이신가요?”처럼 확인 질문을 우선하는 구조를 강화할 계획입니다.

### 20. 예를 들어 “그거 찾아줘”, “아까 본 거 다시 틀어줘” 같은 맥락 의존 명령도 가능한가요?
지금은 제한적입니다. 세션 정보는 일부 남기지만, 진짜 대화 맥락 기반 참조는 아직 본격 구현 전 단계입니다.

### 21. 고령자가 사투리, 불완전한 문장, 반복 표현을 사용하면 어떻게 대응하나요?
현재는 STT 후 정규화와 intent 분류를 통해 일부 보정합니다. 다만 이런 발화를 더 잘 처리하기 위해 한국어 특화 모델과 constraint-first 구조를 함께 고도화하려는 상태입니다.

### 22. 한국어와 일본어가 섞여 들어오는 경우도 고려하고 있나요?
네, 고려하고 있습니다. 아직 완전히 안정적이지는 않지만 언어 힌트, STT 언어 선택, 모델 라우팅을 분리해서 점진적으로 대응 중입니다.

### 23. 사용자가 잘못된 명령을 말했을 때 취소하거나 되돌리는 기능이 있나요?
현재는 자동 취소보다는 재입력과 재실행 중심입니다. 앞으로는 실행 전 확인, 중단, 되돌리기 같은 안전 제어를 더 강화할 계획입니다.

### 24. 사용자가 명령 결과를 이해하지 못했을 때 다시 설명해주는 기능이 있나요?
결과 요약 팝업과 TTS 안내를 통해 쉽게 다시 설명하는 방향으로 구현 중입니다. 특히 고령자용으로 짧고 쉬운 문장으로 다시 안내하는 흐름을 강화하고 있습니다.

### 25. 음성 입력이 실패했을 때 텍스트 입력으로 전환할 수 있나요?
네, 가능합니다. 현재 UI에는 항상 텍스트 입력 경로를 남겨 두고 있습니다.

## 3. 기술 구조 관련 질문

### 26. 전체 시스템 구조를 한 문장으로 설명하면 무엇인가요?
VisionNavi는 Flutter UI에서 받은 자연어 명령을 FastAPI 오케스트레이터가 해석하고, 적절한 브라우저/데스크톱 실행기에 연결해 실제 작업을 수행하는 구조입니다.

### 27. Flutter 프론트엔드, FastAPI 오케스트레이터, 실행 Agent가 각각 어떤 역할을 하나요?
Flutter는 사용자 입력과 상태 표시를 담당합니다. FastAPI 오케스트레이터는 명령 해석, 라우팅, 검증, 세션 관리 역할을 하고, 실행 Agent는 실제 브라우저나 데스크톱 앱을 조작합니다.

### 28. 왜 프론트엔드에서 직접 브라우저를 조작하지 않고 오케스트레이터를 거치나요?
명령 해석, 안전 검증, 로그, fallback, 세션 추적을 한곳에서 관리하기 위해서입니다. 프론트가 직접 조작하면 구조가 단순해 보일 수는 있지만, 복구와 확장이 어려워집니다.

### 29. VisionNavi의 핵심은 LLM인가요, 자동화 엔진인가요, 아니면 오케스트레이터인가요?
핵심은 오케스트레이터입니다. LLM과 자동화 엔진은 중요하지만, 이 둘을 목적에 맞게 연결하고 검증하는 오케스트레이션 레이어가 프로젝트의 중심입니다.

### 30. “AI Agent”라고 부르는 근거는 무엇인가요?
단순 답변 생성이 아니라, 관측하고 판단하고 실행하고 검증하는 반복 구조를 가지기 때문입니다. 아직 모든 시나리오가 완전 자율형은 아니지만, agent 구조로 진화하는 중입니다.

### 31. 단순 매크로 자동화와 VisionNavi의 자동화는 무엇이 다른가요?
매크로는 미리 정한 순서를 반복하는 데 강합니다. VisionNavi는 자연어 명령 해석과 화면 상태 반영을 통해 더 유연하게 목표를 수행하려는 구조입니다.

### 32. 현재 구조에서 LLM은 어느 단계에 개입하나요?
명령 정규화, intent 보조 판단, 일부 plan 생성, popup summary, 일부 next-action 판단에 개입합니다. 다만 길찾기 같은 일부 시나리오는 아직 deterministic fallback 비중이 큽니다.

### 33. 사용자의 자연어 명령이 실제 실행 가능한 명령으로 바뀌는 과정은 어떻게 되나요?
입력 문장을 정규화하고, task domain과 intent를 분류한 뒤, 위험도와 target을 정리합니다. 이 결과가 CanonicalCommand가 되고, 이후 실제 실행기로 전달됩니다.

### 34. 문서에 나오는 CanonicalCommand는 무엇인가요?
사용자 명령을 시스템이 실행 가능한 형태로 정리한 표준 객체입니다. input mode, raw/normalized text, intent, risk, target, notes, constraints 같은 필드를 담습니다.

### 35. CommandConstraint는 왜 필요한가요?
LLM이 검색엔진, 검색어, 언어를 임의로 바꾸는 문제를 막기 위해서입니다. 예를 들어 “네이버에서”라고 했는데 구글로 바뀌는 것을 실행 전에 차단하거나 보정하는 역할을 합니다.

### 36. “네이버에서 검색해줘”라고 했는데 구글로 검색되는 문제를 어떻게 막나요?
현재는 provider를 별도로 추출해 CommandConstraint에 넣고, 실행 전 validator가 mismatch를 검사합니다. external browser도 cross-provider fallback을 허용하지 않도록 정책을 고정하고 있습니다.

### 37. 검색어가 LLM에 의해 임의로 바뀌는 문제는 어떻게 방지하나요?
원문 기반 query_text를 추출하고, 실행 전후에 query drift를 검증합니다. 또한 task prompt에 “검색어를 바꾸지 말 것”을 명시하고, 결과 평가에서도 query preservation을 측정합니다.

### 38. 지도 검색에서 출발지와 도착지가 뒤섞이는 문제는 어떻게 해결하나요?
길찾기 intent에서는 일반 query가 아니라 origin, destination, transport_mode, route_kind를 슬롯으로 분리합니다. 그리고 slot_rewritten 검증으로 뒤섞임을 별도 실패 유형으로 잡고 있습니다.

### 39. intent router는 어떤 기준으로 작업 유형을 분류하나요?
문장 안의 키워드, 도메인 표현, 앱명, 지도/검색/파일 관련 신호를 기준으로 1차 rule 분류를 합니다. 필요한 경우 LLM 보조를 쓰지만, 빠른 rule 경로도 유지하고 있습니다.

### 40. 현재 지원하는 intent에는 어떤 것들이 있나요?
대표적으로 `search_and_read`, `find_map_route`, `open_notepad_and_type`, `inspect_workspace_files`, `change_system_setting`, `general_assistance`가 있습니다.

### 41. search_and_read, find_map_route, open_notepad_and_type 같은 intent는 각각 어떤 실행 경로를 가집니까?
`search_and_read`는 external browser agent 우선 경로를 탑니다. `find_map_route`는 현재 internal browser baseline 비중이 크고, `open_notepad_and_type`는 internal과 external desktop 경로를 모두 실험 중입니다.

### 42. external-first 전략이란 무엇인가요?
신규 발전 축을 내부 하드코딩 실행기보다 외부 agent runtime 중심으로 가져가는 전략입니다. internal은 fallback과 baseline 용도로 유지하고, 주 확장 방향은 external로 두는 것입니다.

### 43. internal executor와 external agent adapter는 각각 어떤 장단점이 있나요?
internal은 재현성과 속도가 좋지만 범용성이 낮습니다. external은 범용성과 장기 확장성이 좋지만, 속도와 안정성이 아직 더 불안정합니다.

### 44. BrowserUse류 외부 브라우저 Agent를 도입하려는 이유는 무엇인가요?
웹사이트 구조가 바뀌어도 더 범용적으로 대응할 가능성이 있기 때문입니다. 우리가 모든 브라우저 자동화를 직접 규칙으로 구현하는 한계를 줄이는 것이 목적입니다.

### 45. UI-TARS 같은 Desktop Agent를 연결하려는 이유는 무엇인가요?
향후 메모장뿐 아니라 다양한 서드파티 Windows 앱까지 확장하기 위해서입니다. 범용 데스크톱 작업으로 가려면 화면 기반 agent 실험이 필요하다고 봤습니다.

## 4. 개발능력 관련 질문

### 46. 현재까지 실제로 구현된 기능과 아직 설계 단계인 기능을 구분해서 설명해 주세요.
실제로 구현된 것은 검색/읽기, 메모장 작성, STT/TTS, wakeword 실험, 고령자 친화 UI입니다. 설계 또는 고도화 단계인 것은 복지 정보 전용 검증 흐름, 안정적인 일본어 wakeword, 범용 external desktop 확대입니다.

### 47. 텍스트 명령 기반 실행은 어느 정도까지 동작하나요?
현재는 검색/읽기와 메모장 작성이 대표 시나리오입니다. 길찾기도 일부 동작하지만 안정성은 아직 더 개선이 필요합니다.

### 48. 음성 입력 기능은 실제 STT와 연동되어 있나요?
네, 실제 STT와 연동되어 있습니다. faster-whisper 기반 로컬 STT와 실시간/파일 전사 흐름을 연결해 두었습니다.

### 49. 웨이크워드 기능은 현재 안정적으로 동작하나요?
한국어는 비교적 실험이 진행됐지만, 일본어는 아직 안정화가 더 필요합니다. 따라서 “구현은 되어 있지만 아직 개선 중”이라고 답하는 것이 가장 정확합니다.

### 50. TTS 안내 기능은 어떤 방식으로 구현되어 있나요?
현재는 별도 상시 TTS worker를 두고, 안내 문장을 요청하면 즉시 합성·재생하도록 구성했습니다. 한국어/일본어 음성 선택도 실험 중입니다.

### 51. faster-whisper를 STT에 사용하는 이유는 무엇인가요?
로컬 실행이 가능하고, 한국어·일본어 다국어 지원이 가능하며, 품질과 속도 균형이 좋기 때문입니다. 또한 GPU가 불안정할 때 CPU fallback을 둘 수 있는 점도 장점입니다.

### 52. livekit-wakeword를 웨이크워드에 사용하는 이유는 무엇인가요?
오픈소스 기반으로 웨이크워드 모델 학습과 런타임 연결이 가능하기 때문입니다. 한국어·일본어 호출어를 직접 다루기 위한 실험 플랫폼으로 적합했습니다.

### 53. Edge TTS를 사용하는 이유는 무엇인가요?
초기에는 로컬 TTS 후보도 검토했지만, 중간 구현 단계에서는 응답 속도와 안정성이 더 중요했습니다. 현재는 Edge TTS를 우선 사용해 한국어·일본어 안내 음성을 빠르게 제공하고, 로컬 TTS는 추후 안정성이 확보된 뒤 다시 비교할 계획입니다.

### 54. Ollama 기반 로컬 LLM을 사용하는 이유는 무엇인가요?
개인정보 보호와 로컬 실험 편의성이 크기 때문입니다. 또한 인터넷 의존도를 줄이고, 모델 교체와 비교 실험을 빠르게 할 수 있습니다.

### 55. 현재 사용하는 LLM 모델이 고령자 대상 한국어·일본어 서비스에 충분하다고 보나요?
아직 충분하다고 보지는 않습니다. 그래서 언어별 메인 모델 분리와 constraint-first 구조 강화를 다음 우선 과제로 두고 있습니다.

### 56. Qwen 계열 모델 사용 시 중국어로 튀는 문제는 어떻게 방지하나요?
단순히 모델만 믿지 않고 provider, query, language 제약을 별도로 검증합니다. 즉, 구조적으로 drift를 막는 것이 우선이고 모델 교체는 그 다음 단계로 보고 있습니다.

### 57. 로컬 LLM을 쓰면 개인정보 보호에는 유리하지만 성능은 떨어질 수 있는데, 이 균형을 어떻게 보고 있나요?
중요한 것은 모든 답을 로컬 LLM이 직접 생성하게 하지 않는 것입니다. 사실성 중요한 작업은 검색/API 기반으로 분리하고, 로컬 LLM은 해석·요약·안내 역할에 더 집중시키는 방향으로 균형을 맞추고 있습니다.

### 58. 실제 서비스화할 경우에도 로컬 실행 구조를 유지할 계획인가요?
초기에는 로컬 중심이 맞다고 봅니다. 다만 장기적으로는 민감 정보는 로컬, 고성능 모델은 서버를 쓰는 하이브리드 구조도 고려할 수 있습니다.

### 59. 서버 기반으로 전환한다면 개인정보 보호 문제는 어떻게 해결할 건가요?
민감 데이터 비식별화, 최소 전송, 동의 기반 수집, 로그 보존 최소화가 필요합니다. 또한 복지·의료 정보 같은 민감 질의는 가능한 로컬 또는 신뢰 가능한 서버로 제한해야 합니다.

### 60. 브라우저 자동화에서 Playwright를 사용하는 이유는 무엇인가요?
브라우저 조작, 대기, DOM 접근, 디버깅 측면에서 안정적인 자동화 프레임워크이기 때문입니다. 실제 사용자 브라우저와 연결하는 CDP 기반 흐름도 지원하기 좋습니다.

### 61. Windows 앱 자동화에서 pywinauto나 UIA를 사용하는 이유는 무엇인가요?
Windows 데스크톱 앱 자동화에서 비교적 널리 쓰이는 안정적인 방식이기 때문입니다. 메모장 같은 기본 앱을 빠르게 검증하기에도 적합했습니다.

### 62. 자동화 도중 화면 구성이 바뀌면 어떻게 복구하나요?
현재는 일부는 deterministic fallback과 재시도로 복구하고, 일부는 external agent 실험으로 대응합니다. 완전한 자율 복구는 아직 진행 중인 과제입니다.

### 63. 실행 성공 여부는 어떻게 검증하나요?
브라우저는 최종 도메인, query token, 요약 언어, 결과 페이지 일치 여부를 봅니다. 데스크톱은 저장 파일 내용과 기대 텍스트 일치 여부 같은 검증을 사용합니다.

### 64. 현재 오류 로그나 세션 trace는 어떤 방식으로 기록되나요?
세션 단위로 canonical review, agent trace, event timeline, execution summary가 남습니다. 실패 원인과 fallback 여부도 함께 저장됩니다.

### 65. 실패 사례를 수집해서 성능 개선에 활용할 계획이 있나요?
네, 그게 현재 개선의 핵심입니다. 특히 drift, fallback, 속도, 실패 taxonomy를 수집해 구조와 모델을 같이 보정하고 있습니다.

## 5. 구현 및 유용성 관련 질문

### 66. 중간발표 시연에서 반드시 보여줘야 할 핵심 장면은 무엇인가요?
자연어 입력 후 실제 브라우저나 메모장이 움직이는 장면입니다. 즉, “AI가 답만 하는 것이 아니라 실제 작업을 수행한다”는 점이 드러나야 합니다.

### 67. 시연 기능이 고령자 문제 해결과 직접적으로 연결되는 이유는 무엇인가요?
검색어 입력, 버튼 클릭, 앱 실행 같은 중간 절차를 대신해 주기 때문입니다. 고령자에게 부담이 큰 디지털 조작 단계를 줄이는 것이 곧 문제 해결과 연결됩니다.

### 68. 시연이 단순 검색 자동화처럼 보이지 않게 하려면 어떤 설명이 필요할까요?
“답변”이 아니라 “실행”이라는 점을 강조해야 합니다. 사용자가 원하는 정보나 작업을 실제 화면 조작으로 끝까지 수행해 주는 흐름이라는 설명이 필요합니다.

### 69. 사용자가 실제로 “말만 하면 된다”는 경험을 느낄 수 있나요?
현재는 상당 부분 가능합니다. 다만 완전한 생활 서비스 수준의 안정성은 아직 개선 중이라, “말하면 실행이 시작되는 경험”까지는 보여줄 수 있다고 답하는 것이 적절합니다.

### 70. 현재 UI는 고령자가 보기 쉬운 구조인가요?
초기 개발자 UI에서 벗어나 고령자 친화형 UI로 많이 개편했습니다. 큰 말하기 버튼, 단순한 문구, 자주 하는 작업 카드, 설정 단순화가 반영돼 있습니다.

### 71. 큰 글씨, 고대비, 단순 버튼 외에 어떤 접근성 요소가 있나요?
상태 문구 단순화, 생활형 표현, TTS 안내, 텍스트 입력 대체 경로, 진행 상태 가시화가 있습니다. 즉, 시각뿐 아니라 인지 부담도 줄이려는 방향입니다.

### 72. 고령자가 시스템이 현재 듣는 중인지, 처리 중인지, 완료했는지 쉽게 알 수 있나요?
그 점을 위해 phase 기반 상태 문구를 정리하고 있습니다. 아직 완벽하지는 않지만, 듣는 중·이해 중·실행 중·완료 상태를 쉽게 보이도록 계속 수정 중입니다.

### 73. 오류가 발생했을 때 고령자에게 어떤 문장으로 안내하나요?
기술 용어 대신 “잠시 후 다시 시도해 주세요”, “입력 장치를 확인해 주세요”처럼 생활형 문구로 바꾸고 있습니다. 디버그 정보는 사용자 모드에서 숨기는 방향입니다.

### 74. 사용자가 기다리는 동안 진행 상태를 어떻게 보여주나요?
메인 히어로 문구와 버튼 상태, 팝업, 일부 TTS로 진행 중임을 안내합니다. 사용자가 “멈췄다”고 느끼지 않게 하는 것이 목표입니다.

### 75. 결과 안내는 텍스트와 음성 중 어느 쪽을 중심으로 하나요?
둘 다 사용하지만, 고령자 대상이라는 점에서 음성 안내 비중을 점점 높일 계획입니다. 현재는 결과 팝업과 TTS를 함께 쓰고 있습니다.

### 76. 사용자가 결과를 신뢰할 수 있도록 어떤 확인 과정을 제공하나요?
가능하면 실제 사이트 방문 결과와 출처 기반 요약을 보여주는 방향입니다. 또한 사실성이 중요한 정보는 모델 직답보다 검증된 정보 흐름으로 분리하려고 합니다.

### 77. 잘못된 웹사이트나 광고를 클릭할 위험은 어떻게 줄이나요?
provider, domain, query, language를 검증하고 off-target navigation을 실패로 분류합니다. 아직 완전하지는 않지만 구조적으로 막는 장치를 강화하고 있습니다.

### 78. 자동화가 개인정보 입력 화면에 도달하면 어떻게 처리하나요?
민감한 입력은 자동 진행하지 않고 확인 절차를 두는 방향입니다. 향후 안전 설정과 confirmation 정책을 더 강화할 예정입니다.

### 79. 결제, 로그인, 개인정보 입력 같은 민감 작업은 자동으로 실행하나요?
아니요, 기본적으로는 자동 실행 대상이 아닙니다. 현재도 위험도 분류와 확인 절차를 두고 있고, 장기적으로는 더 엄격히 제한할 계획입니다.

### 80. 고령자가 실수로 위험한 명령을 내렸을 때 확인 절차가 있나요?
있습니다. 위험도와 intent에 따라 confirmation이 필요한 구조를 두고 있습니다.

## 6. 데이터·보안·개인정보 관련 질문

### 81. 사용자의 음성 데이터는 저장되나요?
현재는 기능 수행을 위한 임시 처리 중심입니다. 장기 보관은 기본값이 아니며, 필요 시 저장 정책을 명확히 분리해야 합니다.

### 82. 음성 명령 로그는 어디에 저장되며, 어떤 목적으로 사용되나요?
세션 trace와 실행 결과는 디버깅과 개선 목적으로 남길 수 있습니다. 다만 실제 서비스에서는 저장 범위와 보관 기간을 엄격하게 설계해야 합니다.

### 83. 개인정보가 포함된 명령이 들어오면 어떻게 처리하나요?
현재도 민감 작업은 자동 실행을 제한하는 방향입니다. 앞으로는 개인정보 키워드 탐지와 추가 확인 흐름을 더 넣을 계획입니다.

### 84. 가족 연락처, 주소, 병원 정보 같은 민감 데이터는 어떻게 보호하나요?
가능한 로컬 처리와 최소 저장 원칙을 우선합니다. 실제 서비스 단계에서는 암호화와 접근 통제도 필요합니다.

### 85. 로컬 실행 구조가 개인정보 보호에 어떤 장점이 있나요?
음성, 명령, 실행 로그가 외부 API로 무조건 전송되지 않는다는 점입니다. 고령자 서비스에서 이 점은 신뢰 확보에 중요합니다.

### 86. 반대로 로컬 실행 구조의 한계는 무엇인가요?
모델 성능과 속도가 서버형보다 제한될 수 있습니다. 장치 성능, 설치 관리, 모델 업데이트 부담도 있습니다.

### 87. 외부 LLM API를 사용할 경우 개인정보 유출 가능성은 어떻게 통제하나요?
민감 정보 마스킹, 최소 전송, 사용자 동의, 로깅 제한이 필요합니다. 가능하면 사실성 높은 공공정보 탐색은 API보다 검색/RAG 기반으로 가는 것이 낫습니다.

### 88. 사용자가 원하지 않는 자동 조작을 막는 안전장치가 있나요?
위험도 분류, confirmation, backend 정책, domain 검증 같은 구조가 그 역할을 합니다. 아직 더 강화할 부분은 있지만 방향은 분명합니다.

### 89. 웹 자동화 중 악성 사이트나 피싱 페이지를 만나면 어떻게 대응하나요?
현재는 provider/domain drift를 검출하고 실패로 분류합니다. 향후 화이트리스트와 민감 페이지 차단 같은 정책을 더 넣어야 합니다.

### 90. 시스템이 잘못된 정보를 안내했을 때 책임 문제는 어떻게 보나요?
그래서 복지·의료·공공정보 같은 사실성 높은 영역은 LLM 직답보다 검증된 출처 기반 요약으로 분리하려고 합니다. 불확실할 때는 단정하지 않는 답변 정책도 중요합니다.

## 7. 파급성 관련 질문

### 91. 이 프로젝트가 실제 사업화될 가능성이 있다고 보나요?
있다고 봅니다. 다만 개인 단독 판매보다 복지기관, 지자체, 시니어 센터와 연결하는 방식이 더 현실적입니다.

### 92. 사업화한다면 주요 고객은 개인 고령자인가요, 복지관·지자체·요양기관인가요?
초기에는 B2B2C나 B2G가 더 적합하다고 봅니다. 기관 도입을 통해 실제 사용자 접점을 만드는 방식이 현실적입니다.

### 93. B2C보다 B2G 또는 B2B2C 모델이 더 적합하지 않나요?
네, 그렇게 보고 있습니다. 고령자 대상 서비스는 신뢰, 지원, 교육이 함께 가야 해서 기관 연계형이 더 적합합니다.

### 94. 복지관, 주민센터, 요양시설에 설치하는 방식도 고려하고 있나요?
고려하고 있습니다. 공용 PC나 안내 데스크 환경에 적용하는 시나리오도 충분히 가능합니다.

### 95. 한국뿐 아니라 일본 고령자까지 타겟으로 확장하려는 이유는 무엇인가요?
고령화 사회라는 공통 문제가 있고, 디지털 격차 해소라는 사회적 가치도 공유되기 때문입니다. 또한 다국어 지원을 통해 확장 가능성을 보여줄 수 있습니다.

### 96. 일본 시장에 적용하려면 어떤 부분을 추가로 바꿔야 하나요?
언어뿐 아니라 검색 서비스, 존댓말 표현, TTS 음색, wakeword, 사용자 안내 문구를 일본 사용자 습관에 맞게 바꿔야 합니다.

### 97. 고령자 디지털 교육 프로그램과 연계할 수 있나요?
가능합니다. VisionNavi는 단순 대행뿐 아니라 “무엇을 어떻게 하는지”를 같이 보여줄 수 있어 교육 보조 도구로도 활용 가능합니다.

### 98. 가족 돌봄 부담 감소라는 효과를 어떻게 측정할 수 있나요?
보호자 도움 요청 횟수 감소, 작업 성공률 증가, 재시도 횟수 감소 같은 지표로 간접 측정할 수 있습니다.

### 99. 정서적 안정이나 사회적 고립 완화 효과는 어떻게 검증할 계획인가요?
직접적인 의료적 효과를 주장하기보다는, 정보 접근성 향상과 디지털 자립감 증가를 사용자 조사로 측정하는 방식이 더 적절합니다.

### 100. 실제 사용자 테스트를 한다면 어떤 지표를 볼 건가요?
명령 인식 성공률, 작업 완료율, 평균 완료 시간, 재시도 횟수, 사용자 만족도, 보호자 개입 빈도를 볼 계획입니다.

### 101. 사용 성공률, 명령 인식률, 작업 완료 시간, 재시도 횟수 중 무엇이 가장 중요한 지표인가요?
가장 중요한 것은 작업 완료율입니다. 고령자 서비스에서는 “결국 해냈는가”가 핵심이고, 그 다음이 시간과 재시도 횟수라고 봅니다.

### 102. 이 프로젝트가 기존 AI 스피커와 경쟁한다면 차별점은 무엇인가요?
AI 스피커는 주로 질의응답과 IoT 중심입니다. VisionNavi는 실제 웹사이트와 PC 앱을 대신 조작하는 작업 수행형 서비스라는 점이 다릅니다.

### 103. 공공기관 지원사업에 제출한다면 가장 강조해야 할 사회적 가치는 무엇인가요?
고령자의 디지털 접근성 향상과 정보 격차 완화입니다. 특히 복지·정보·생활 서비스 접근성을 높인다는 점을 강조할 수 있습니다.

### 104. 유지보수 비용이 많이 들 가능성은 없나요?
있습니다. 특히 웹 자동화는 사이트 변화 대응 비용이 큽니다. 그래서 범용 agent와 constraint 구조를 결합해 유지보수 부담을 낮추는 방향으로 가고 있습니다.

### 105. 웹사이트 UI가 바뀔 때마다 시스템을 수정해야 한다면 사업성이 떨어지지 않나요?
그 위험이 있기 때문에 external-first와 범용 agent 통합을 추진하는 것입니다. 완전 해결은 어렵지만, 고정 규칙만 쓰는 방식보다 장기 유지보수성이 낫다고 판단합니다.

## 8. 팀워크 관련 질문

### 106. 팀원별 역할 분담은 어떻게 되어 있나요?
기획, 발표자료, UI 설계, 시스템 구현, 문서화로 나누어 분담하는 방식입니다. 실제 발표에서는 본인 담당 영역을 명확히 말하는 것이 중요합니다.

### 107. 발표자료, 기획서, 시스템 개발, UI 설계, 문서화는 각각 누가 담당했나요?
이 질문은 팀별 실제 역할 분담에 맞춰 답해야 합니다. 다만 “기획-개발-UI-문서화를 병행하며 서로 피드백했다”는 구조로 정리하면 좋습니다.

### 108. 팀원 간 기술 수준 차이는 어떻게 보완했나요?
문서화와 읽기 순서 가이드를 통해 맥락 공유를 강화했습니다. 또한 기능 단위를 작게 쪼개 작업 부담을 나누는 방식으로 보완했습니다.

### 109. 개발 중 의견 충돌이 있었다면 어떻게 조율했나요?
최종 목표와 현재 구현 가능 범위를 분리해서 조율했습니다. 예를 들어 external-first 전략도 “장기 방향”과 “현재 fallback 유지”를 나눠 정리하며 합의했습니다.

### 110. 팀원 탈퇴나 인원 변화가 있었다면 프로젝트 진행에 어떤 영향을 주었나요?
범위 조정의 필요성이 커졌을 것입니다. 그 경우 핵심 시나리오를 우선순위로 재정렬해 완성 가능한 범위에 집중했다고 답하는 것이 좋습니다.

### 111. 현재 남은 팀원 기준으로 개발 범위를 조정했나요?
네, 조정했습니다. 검색/읽기, 메모장 자동화, UI, 음성 흐름 같은 핵심 시나리오에 집중하는 식으로 범위를 관리하고 있습니다.

### 112. 일정 관리와 작업 분배는 어떤 도구나 방식으로 하고 있나요?
문서와 GitHub 중심으로 관리하고, 기능 단위로 정리해 순차적으로 진행하고 있습니다. 중요한 것은 누가 무엇을 언제까지 맡는지 분명히 하는 것입니다.

### 113. GitHub나 문서 관리 체계는 어떻게 운영하고 있나요?
기술 가이드, 개발 히스토리, 발표 문서, 읽기 순서 문서를 별도로 두고 있습니다. 즉, 단순 코드 저장소가 아니라 온보딩 가능한 구조로 정리하고 있습니다.

### 114. 프로젝트 문서가 많은데, 팀원이 길을 잃지 않도록 어떤 읽기 순서를 정했나요?
MustRead 폴더와 reading-order 문서를 통해 우선 읽을 문서를 정리했습니다. 처음 보는 사람도 전체 구조를 빠르게 이해할 수 있게 만들었습니다.

### 115. 새 팀원이 합류하면 어떤 문서부터 읽게 할 건가요?
먼저 전체 시스템 가이드, 그 다음 읽기 순서 가이드, 그 다음 발표/Q&A 문서를 보게 할 것입니다. 즉, 구조 이해 → 코드 위치 이해 → 발표 맥락 이해 순서입니다.

## 9. 발표 및 시연 관련 질문

### 116. 5분 발표 안에서 가장 강조해야 할 메시지는 무엇인가요?
“고령자의 디지털 작업을 단순 답변이 아니라 실제 실행으로 돕는 시스템”이라는 점입니다. 그리고 “지금 어디까지 구현됐는가”를 정직하게 보여주는 것이 중요합니다.

### 117. 시연 영상이 3분 이상이면 발표 설명 시간이 부족하지 않나요?
맞습니다. 그래서 발표 본문은 2분 안팎, 시연은 핵심 장면 위주로 짧게 가져가는 것이 좋습니다.

### 118. 발표에서 기술 설명과 문제 설명의 비중은 어떻게 조절했나요?
문제와 필요성을 먼저 60%, 기술은 40% 정도로 보는 것이 적절합니다. 중간발표에서는 “왜 필요한가”가 더 중요합니다.

### 119. 심사위원이 기술자가 아닐 경우에도 이해할 수 있게 설명할 수 있나요?
네, 가능합니다. “명령을 이해하고 실제 브라우저와 PC를 대신 조작한다”는 한 문장 구조로 설명하면 비기술자도 이해하기 쉽습니다.

### 120. 발표자료의 “Navi의 해결 방식” 흐름도는 실제 구현 구조와 일치하나요?
핵심 흐름은 일치합니다. 다만 세부 구현은 internal/external, fallback, validator처럼 더 복잡하므로 Q&A에서 보충 설명하면 됩니다.

### 121. 발표자료에서는 “오픈소스 AI Agent”라고 표현했는데, 어떤 오픈소스를 활용했나요?
BrowserUse류 브라우저 agent, UI-TARS류 데스크톱 agent, faster-whisper, livekit-wakeword, Edge TTS 등을 활용하고 있습니다.

### 122. 발표자료에서 “개인정보 보호 중심의 로컬 실행”이라고 했는데, 실제 구현도 완전히 로컬인가요?
핵심 음성, TTS, 로컬 LLM, 오케스트레이터는 로컬 중심입니다. 다만 웹 검색 자체는 외부 사이트와 상호작용하므로 “완전 폐쇄형 로컬”은 아닙니다.

### 123. 시연 중 실패하면 어떻게 대응할 계획인가요?
사전에 녹화된 영상과 대체 시나리오를 준비해야 합니다. 그리고 실패 자체도 현재 한계와 다음 개선 포인트로 설명할 수 있어야 합니다.

### 124. 인터넷 연결 문제나 모델 응답 지연이 발생하면 대체 시나리오가 있나요?
텍스트 입력, 로컬 메모장 시나리오, 녹화 영상이 대체 시나리오가 될 수 있습니다. 발표에서는 가장 안정적인 흐름을 우선 시연하는 것이 중요합니다.

### 125. 발표에서 “현재 구현된 것”과 “앞으로 구현할 것”을 명확히 구분할 수 있나요?
반드시 그렇게 해야 합니다. 중간발표에서는 현재 구현을 과장하지 않고, 다음 단계는 계획으로 명확히 구분하는 것이 신뢰를 줍니다.

## 10. 반론·공격형 질문

### 126. 고령자가 정말 이런 AI Agent를 사용할 수 있을까요?
모든 고령자가 바로 사용할 수 있다고 보지는 않습니다. 하지만 복잡한 조작 대신 말하기와 큰 버튼 중심으로 접근성을 높이면 충분히 유용한 사용자층이 있다고 봅니다.

### 127. 음성비서가 이미 많은데 굳이 Navi가 필요한 이유는 무엇인가요?
기존 음성비서는 대답은 잘하지만 실제 PC 작업 수행은 약합니다. VisionNavi는 웹과 데스크톱 작업까지 연결하려는 점이 다릅니다.

### 128. 웹 자동화는 사이트 구조가 바뀌면 쉽게 망가지지 않나요?
맞습니다. 그래서 현재도 큰 한계로 보고 있고, 그 때문에 범용 agent와 constraint-first 구조를 같이 도입하고 있습니다.

### 129. LLM이 잘못 판단하면 오히려 고령자에게 더 큰 혼란을 줄 수 있지 않나요?
그 가능성이 있기 때문에 LLM을 무제한으로 믿지 않고 validator와 confirmation을 두고 있습니다. 특히 사실성 높은 영역은 검색/API 기반으로 분리하려고 합니다.

### 130. 고령자 대상 서비스라면 안정성이 가장 중요한데, AI Agent 방식은 너무 불안정하지 않나요?
현재는 그 한계를 인정합니다. 그래서 중간발표에서는 “완성 서비스”가 아니라 “실행형 구조를 검증 중인 프로토타입”이라고 설명하는 것이 맞습니다.

### 131. 현재 기술 수준으로 실제 고령자에게 제공하기에는 이르지 않나요?
네, 아직은 시범 검증 단계에 가깝습니다. 다만 핵심 시나리오와 사용자 경험 방향은 충분히 검증할 가치가 있다고 봅니다.

### 132. 단순히 유튜브 검색이나 지도 검색을 자동화하는 수준이라면 혁신성이 부족하지 않나요?
단순 기능 하나만 보면 그렇게 보일 수 있습니다. 하지만 자연어 입력, 음성 흐름, 실행 검증, 로컬 AI, 고령자 UX를 통합한 실행형 구조라는 점이 차별점입니다.

### 133. 프로젝트 목표가 너무 크고 구현 범위가 넓지 않나요?
맞습니다. 그래서 현재는 핵심 시나리오를 좁혀서 중간발표 범위를 관리하고 있습니다.

### 134. 음성 인식, LLM, 웹 자동화, TTS를 모두 다루면 완성도가 낮아질 위험은 없나요?
그 위험이 있기 때문에 각 축을 모두 완성하려 하기보다, 연결 구조와 핵심 경험을 우선 검증하고 있습니다.

### 135. 개인정보 보호를 강조하면서 외부 웹 서비스를 자동으로 조작하는 것은 모순 아닌가요?
완전한 모순이라기보다 균형의 문제입니다. 음성과 명령 처리 자체는 로컬 중심으로 하고, 외부 웹 조작은 필요한 최소 범위로 제한하려는 방향입니다.

### 136. 고령자가 실수로 이상한 명령을 하면 시스템이 잘못된 행동을 할 가능성은 없나요?
가능성은 있습니다. 그래서 위험 작업은 확인을 요구하고, provider/query/language drift를 막는 구조를 강화하고 있습니다.

### 137. 실제 치매 고령자에게 사용하게 하는 것은 윤리적으로 조심해야 하지 않나요?
맞습니다. 현재 주 대상도 치매 환자보다는 디지털 취약 고령자입니다. 치매 영역으로 가려면 보호자 동의와 안전성 검증이 훨씬 더 중요합니다.

### 138. 가족이나 보호자의 동의 없이 사용할 수 있나요?
일반적인 디지털 보조 도구로는 가능하겠지만, 민감한 사용자군이나 의료·돌봄 맥락에서는 보호자와 기관 기준을 함께 봐야 합니다.

### 139. 정서적 안정 효과를 기술적으로 증명할 수 있나요?
직접적 증명은 어렵습니다. 대신 자립감, 작업 성공 경험, 보호자 도움 감소 같은 지표를 통해 간접적으로 평가하는 것이 현실적입니다.

### 140. 사회적 고립 완화라는 목표가 너무 추상적이지 않나요?
그 자체만으로는 추상적일 수 있습니다. 그래서 실제 발표에서는 “정보 접근성 향상”, “디지털 자립감 증가”, “복지·여가 서비스 접근”처럼 더 구체적인 가치로 설명하는 것이 좋습니다.

## 11. 답변 준비가 특히 필요한 핵심 질문 10개

### 1. 기존 챗봇이나 AI 스피커와 Navi의 차이점은 무엇인가요?
기존 서비스는 주로 답변과 대화 중심이지만, VisionNavi는 실제 웹과 PC 작업 수행까지 연결하는 점이 다릅니다.

### 2. 현재 실제로 구현된 기능은 어디까지인가요?
검색/읽기, 메모장 작성, STT/TTS, wakeword 실험, 고령자 친화 UI까지 구현되어 있습니다. 다만 길찾기와 일본어 wakeword 안정화는 아직 개선 중입니다.

### 3. 고령자에게 정말 유용하다는 근거는 무엇인가요?
고령자에게 부담이 큰 중간 조작 단계를 줄여 주기 때문입니다. 즉, 검색어 입력과 버튼 클릭 같은 디지털 절차를 대신해 주는 것이 직접적인 효용입니다.

### 4. 음성 명령이 들어오면 내부적으로 어떤 순서로 처리되나요?
STT로 문장을 얻고, 오케스트레이터가 intent와 target을 정리한 뒤, 실행 채널을 선택하고, 실제 브라우저나 데스크톱을 조작하고, 마지막에 결과를 안내합니다.

### 5. LLM이 잘못된 사이트나 검색어로 바꾸는 문제는 어떻게 막나요?
CommandConstraint와 CommandValidator로 provider, query, language를 별도로 고정하고 검증합니다. 즉, LLM 위에 구조적 안전장치를 올리는 방식입니다.

### 6. 개인정보 보호는 어떻게 하나요?
핵심 음성과 명령 처리는 로컬 중심으로 두고, 민감 작업은 자동 실행하지 않는 방향으로 설계합니다. 또한 사실성 높은 정보는 검증된 출처 기반으로 다루려 합니다.

### 7. 시연 기능이 최종 목표와 어떻게 연결되나요?
지금 시연하는 검색과 메모장 자동화는, 장기적으로 더 다양한 웹·데스크톱 작업으로 확장할 agent형 구조의 대표 예시입니다.

### 8. 웹사이트 UI가 바뀌어도 계속 동작할 수 있나요?
완벽하다고 말할 수는 없습니다. 대신 internal 규칙만 쓰지 않고 external agent와 validator 구조를 결합해 변화 대응력을 높이려는 방향입니다.

### 9. 사업화나 실제 적용 가능성은 어떻게 보나요?
개인 판매형보다 복지기관, 지자체, 시니어센터와 연계하는 B2B2C/B2G 모델이 더 현실적이라고 보고 있습니다.

### 10. 중간발표 이후 가장 먼저 개선할 기능은 무엇인가요?
첫째는 provider/query/language drift를 막는 구조 안정화, 둘째는 external browser와 일본어 음성 흐름의 안정화입니다.
